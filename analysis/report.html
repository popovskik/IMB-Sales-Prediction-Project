1. The question

A pizza restaurant wants to plan staffing, inventory, and purchasing ahead of demand. Two operational questions drive this study, using one full year (2015) of order data:

  • Regression — how much will we take today? Predict Daily_Revenue, the sum of price × quantity across all of a day’s orders.
  • Classification — will today be busy? Predict High_Demand_Day: a day whose order count exceeds the year’s mean daily order count.

Both targets are aggregated to one row per calendar day, and both models use only features derived from the date — so a prediction can be made for any future day with no other information.

2. The data

The Maven Pizza Place Sales dataset is four linked tables: orders, order_details, pizzas, and pizza_types (~48,620 line items across ~21,350 orders). We join them, compute line revenue, and aggregate to a daily series.

4. Feature engineering (leakage-safe)

Two feature sets are built from the date (src/features.py):

  • Served set — calendar parts (day_of_week, month, week_of_year, day_of_year, quarter), boolean flags (is_weekend, is_month_start, is_month_end), and cyclical sin/cos encodings of weekday and month (so December sits next to January). These are fully determined by the date, so the deployed API can compute them for any day.
  • Full set — adds past-only lag/rolling features (revenue_lag_1, revenue_lag_7, rolling_revenue_7d, …). Each uses shift, and the rolling mean uses shift(1) first so today never leaks into its own feature. Used only for an offline comparison, since a date-only predictor has no “yesterday” at serve time.

Leakage controls. Scalers and the feature selector are fit inside the training split only (sklearn Pipeline), and the data is split chronologically — never shuffled.

5. Models and the honest leaderboard

Each task trains a full ladder — Dummy → linear → Decision Tree → tuned XGBoost — with XGBoost tuned by RandomizedSearchCV over a TimeSeriesSplit (so tuning never sees the future). We report two numbers per model: the TimeSeriesSplit CV score (the fairer estimate) and the Nov–Dec hold-out score.

Code 
def reg_table(rows):
    out = [{"Model": r["model"], "CV R²": round(r["cv_r2"], 3),
            "Test R²": round(r["test"]["r2"], 3), "Test MAE": round(r["test"]["mae"], 1),
            "Test RMSE": round(r["test"]["rmse"], 1), "Train R²": round(r["train_r2"], 3)}
           for r in rows if r["task"] == "regression"]
    return pd.DataFrame(out)

def clf_table(rows):
    out = [{"Model": r["model"], "CV F1": round(r["cv_f1"], 3),
            "Test Acc": round(r["test"]["accuracy"], 3), "Test F1": round(r["test"]["f1"], 3),
            "Test ROC-AUC": (None if r["test"]["roc_auc"] is None else round(r["test"]["roc_auc"], 3))}
           for r in rows if r["task"] == "classification"]
    return pd.DataFrame(out)

5a. The key result: CV and the hold-out disagree — and why

This is the most important finding in the study, and it is a direct consequence of having only one year of data:

  • On TimeSeriesSplit CV, XGBoost is the best regressor (CV R² ≈ 0.21), clearly beating the mean baseline (≈ 0).
  • On the Nov–Dec hold-out, the tree models score near-zero or negative R².

The reason is not that the models are broken — it is that the hold-out months (November, December) are calendar positions the model never saw in training (Jan–Sep). Tree models cannot extrapolate month/week_of_year beyond their training range, so they fail precisely on the unseen tail; linear Ridge, which extrapolates, edges them out on that specific hold-out. With a single year, any future hold-out has this problem. We therefore select the deployed model by the CV estimate (the documented primary metric), giving XGBoost for revenue.

6. Honest evaluation

  • Report test performance, keep the baseline. Every leaderboard row shows hold-out metrics with the Dummy baseline retained. We do not quote training scores as results.
  • Overfitting is shown, not hidden. The depth curve makes the train-vs-CV gap explicit; the deeper trees memorise the training year and generalise worse.
  • Metric choice matters. The majority-class classifier scores a deceptively high F1 (0.60) by predicting “high” for every day (perfect recall, no true negatives). Judged by ROC-AUC — the threshold-independent metric — it sits at 0.50 (random), while Logistic Regression reaches 0.74. We therefore select and judge the classifier by AUC, not F1.
  • What we would and would not trust. The classifier is a useful “busy day?” signal for staffing (AUC ≈ 0.74 from the calendar alone). The revenue regressor explains only a modest share of daily variance — daily revenue is noisy, and a single year limits how far any model can generalise. We would use it for rough planning, not financial commitments, and we would retrain as more years of data accrue.

7. Business interpretation

  • Weekends and Fridays are the demand peaks — the weekly pattern is the strongest, most reliable signal. Staff and stock to it.
  • A simple calendar model already flags busy days well enough to inform shift planning (AUC ≈ 0.74), with no point-of-sale integration required.
  • Revenue forecasting needs more than the calendar. Promotions, weather, and local events likely drive the unexplained variance; capturing them is the clear next step.
  • More history is the highest-value investment. The single-year limitation — not the model choice — is what caps accuracy here.
